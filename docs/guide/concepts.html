Core Concepts

This page explains the foundational ideas behind NeuralFlow: how it runs graphs, where it stores memory, how it integrates with external tools via MCP, and how it accounts for every token spent.

Execution Modes

NeuralFlow supports four execution modes. Each mode determines how the orchestrator traverses the graph and delegates work to agent nodes. Select the mode in the workflow Settings panel or let NeuralFlow choose automatically.

Sequential

The orchestrator performs a topological sort of the graph and executes nodes left-to-right in dependency order. A node runs only after all its upstream predecessors have completed. This is the simplest mode and works for any acyclic graph.

  • Nodes with no dependencies between them run in parallel within the same topological level.
  • The executor is a plain Python asyncio task runner — no additional framework dependency.
  • Use Sequential for pipelines, chains, and most common workflows.

Hierarchical (CrewAI)

A manager AgentNode receives the initial task and dynamically delegates subtasks to worker agents using CrewAI's hierarchical process. Workers report results back to the manager, which synthesises a final answer.

  • Requires at least one node tagged as role: manager in the AgentNode properties.
  • Worker nodes do not need explicit edges — CrewAI discovers them via role declarations.
  • Best suited for research, writing, and complex decomposition tasks where the manager can adaptively decide which workers to invoke.
Note: Hierarchical mode requires the crewai extra to be installed. Run uv sync --extra crewai in packages/sidecar.

State Machine (LangGraph)

The graph is compiled into a LangGraph StateGraph where each node is a graph node and each edge is a transition. Unlike Sequential mode, the State Machine mode supports cycles — a node can loop back to an earlier node based on the output of a RouterNode or a conditional edge.

  • State is persisted as a typed dict across all nodes in a run.
  • Cycles are explicitly supported, enabling retry loops and iterative refinement patterns.
  • Use State Machine for agentic loops, self-correcting pipelines, and any workflow that needs to revisit a node.
Warning: Cyclic graphs in Sequential mode raise a CyclicGraphError at run start. Use State Machine mode for any workflow containing backward edges.

Auto

NeuralFlow inspects the graph shape at run time and picks the most appropriate mode:

  • If the graph is acyclic and contains no manager-role agent → Sequential.
  • If the graph contains a manager-role agent → Hierarchical.
  • If the graph contains cycles → State Machine.

Auto is the default for new workflows. Override it only when NeuralFlow's heuristic picks the wrong mode.

Memory & RAG

NeuralFlow includes a built-in retrieval-augmented generation (RAG) pipeline backed by SQLite and sqlite-vec. No external vector database is required.

MemoryChunk table

Ingested documents are split into chunks and stored in the memory_chunks table in the workspace SQLite database. Each row stores:

  • id — UUID primary key
  • source — original filename or URL
  • content — the raw text of the chunk
  • embedding — 1536-dimensional float32 vector (OpenAI text-embedding-3-small by default)
  • metadata — JSON blob with page number, section heading, and other provenance fields

Ingest documents

Use the POST /api/memory/ingest endpoint or the Ingest button in the Memory panel. Supported file formats: TXT, MD, PDF, and DOCX. The sidecar extracts text, chunks it with a configurable window size (default 512 tokens, 64-token overlap), embeds each chunk, and writes to SQLite. 

curl -X POST http://127.0.0.1:7411/api/memory/ingest \
  -F "file=@research-paper.pdf" \
  -F "chunk_size=512" \
  -F "overlap=64"

Semantic search

At query time, the sidecar embeds the query string and computes cosine similarity against all stored vectors using sqlite-vec's vec_distance_cosine function. The top-k chunks (default k=5) are returned.

If the embedding model is unavailable (e.g., no API key), the search falls back to SQLite's full-text search (FTS5) using keyword matching. Set NEURALFLOW_MEMORY_FALLBACK=fts to force keyword mode.

Use memory in a workflow

Add a ToolNode with tool_name: memory_search and connect it upstream of any AgentNode that needs context. The tool accepts a query field and injects the retrieved chunks into the AgentNode's context window automatically.

MCP Integration

The Model Context Protocol (MCP) is an open standard for exposing tools and resources to LLM agents. NeuralFlow acts as an MCP client and can connect to any compliant MCP server.

Transports

NeuralFlow supports two MCP transport mechanisms:

  • stdio — the sidecar spawns the MCP server as a subprocess and communicates over stdin/stdout. This is the most common option for local servers distributed as npm or pip packages.
  • SSE — the sidecar connects to a long-running MCP server over HTTP Server-Sent Events. Use this for remote or persistent MCP services.

Connection pool

The sidecar maintains a connection pool in packages/sidecar/neuralflow/mcp/. Connections are keyed by the (transport, command/url) pair and reused across nodes within a run. Idle connections are closed after 5 minutes of inactivity.

Tool registry

When a McpNode or AgentNode connects to an MCP server for the first time, the sidecar fetches the server's capability manifest and registers its tools in the global tool registry. Registered MCP tools appear alongside built-in tools in the AgentNode's tool picker in the Properties panel.

MCP tools in AgentNode

To let an AgentNode call MCP tools during inference, add the tool names to the tools list in the AgentNode's properties. The sidecar converts each MCP tool schema into the provider-specific function-calling format (OpenAI, Anthropic, etc.) and handles round-trips transparently.

Tip: Use the filesystem MCP server to give agents read/write access to local files without writing custom tool code: 
# McpNode — transport: stdio
command: npx -y @modelcontextprotocol/server-filesystem /home/user/projects
tool_name: read_file

Human-in-the-Loop

NeuralFlow's HumanNode implements a pause-and-resume pattern that lets an operator review intermediate results and provide input or approval before execution continues.

How the pause works

  1. The executor reaches a HumanNode and immediately suspends the run.
  2. The sidecar emits an awaiting_input SSE event on the run's stream (GET /api/runs/{id}/stream) with the node's prompt field.
  3. The frontend receives the event, highlights the paused HumanNode on the canvas, and opens a modal dialog showing the prompt.
  4. The operator types a response (or clicks Approve/Reject for approval type) and submits.
  5. The frontend calls POST /api/runs/{id}/resume with the submitted text.
  6. The sidecar unblocks the run, injects the response as the HumanNode's output, and continues execution from the next node in the graph.

Timeout behaviour

If timeout_s is set to a non-zero value on the HumanNode, the run is automatically cancelled with a timed_out status if no resume call arrives within that duration. Set timeout_s: 0 (the default) to wait indefinitely.

Note: Paused runs persist across sidecar restarts. The pause state is stored in the SQLite runs table as status awaiting_input. On restart, the sidecar re-registers the pending SSE subscription so the frontend can reconnect seamlessly.

Cost Tracking

Every LLM inference call made by an AgentNode records token counts and cost in USD. Cost data is used to populate the Cost Dashboard and to enforce optional cost alerts.

How costs are recorded

NeuralFlow uses LiteLLM's in-process cost calculation. After each completion, LiteLLM returns a UsageData object containing:

  • prompt_tokens — tokens in the input (system prompt + context + user message)
  • completion_tokens — tokens in the model's output
  • cost_usd — computed from LiteLLM's price table for the model

These values are written to the node_runs table and rolled up into the parent runs record.

Analytics API

Retrieve aggregated cost data via the Analytics API. The endpoint supports filtering by time window and workflow. 

GET /api/analytics/costs?days=30&workflow_id=<uuid>

The response includes a per-day breakdown, a per-model breakdown, and a per-workflow breakdown. All costs are in USD.

Cost alerts

Set a cost_alert_usd threshold in the workspace settings. When a single run exceeds the threshold, the sidecar emits a cost_alert SSE event and the frontend displays a warning banner. The run is not cancelled — alerts are advisory only.

Plugin System

NeuralFlow's plugin system lets you add custom node types and tools without modifying the core codebase. Plugins are standard Python packages that declare a neuralflow_plugins entry point group.

Create a plugin package

A minimal plugin contains a pyproject.toml entry point declaration and a Python module that exports a register() function. 

# pyproject.toml
[project.entry-points."neuralflow_plugins"]
my_plugin = "my_plugin.plugin:register"
# my_plugin/plugin.py
from neuralflow.plugins import PluginManifest, register_tool

def register(manifest: PluginManifest) -> None:
    manifest.add_tool(
        name="my_tool",
        description="Does something useful.",
        parameters={"query": {"type": "string"}},
        handler=my_tool_handler,
    )

Install a plugin

cd packages/sidecar
uv pip install /path/to/my-plugin

PluginManifest API

The PluginManifest object passed to register() exposes two registration methods:

  • manifest.add_tool(name, description, parameters, handler) — registers a callable as a built-in tool. The tool appears in the ToolNode picker and can be attached to AgentNodes.
  • manifest.add_node_type(type_id, schema, executor_class) — registers a custom node type. The type appears in the node palette on the canvas and its executor class is called during runs.

Hot reload

In development, the sidecar watches the plugin entry points at startup and re-imports changed modules when the --reload flag is active (uvicorn hot reload). In production builds, restart the sidecar after installing or updating a plugin.

Note: Plugins run in the same process as the sidecar. A plugin that raises an unhandled exception during register() will prevent the sidecar from starting. Check the sidecar log for the full traceback.
